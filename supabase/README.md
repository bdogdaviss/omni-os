# Database schema & migrations

The source of truth for the database is **`supabase/migrations/`**. Right now
that directory is empty — the live schema (15 tables, RLS, and the unique
constraints the app's duplicate detection relies on) currently lives only in the
Supabase project, not in this repo. The steps below capture it once, after
which a fresh clone can rebuild the database from version control.

`config.toml` was generated by `supabase init` and is safe to commit — every
secret in it is an `env(...)` reference, not a literal.

## One-time: capture the live schema

`supabase db pull` connects directly to Postgres, so it needs the **database
password** (not the anon/publishable key in `.env.local`). Run these yourself —
the password isn't in the repo:

```bash
# 1. Authenticate the CLI (opens a browser; or set SUPABASE_ACCESS_TOKEN)
supabase login

# 2. Link this repo to the project (prompts for the DB password)
supabase link --project-ref edwfaaisgtnbfbsaymca

# 3. Snapshot the live schema into supabase/migrations/<timestamp>_remote_schema.sql
supabase db pull

# 4. Review the generated file, then commit it
git add supabase/migrations && git commit -m "Capture production schema as initial migration"
```

After the pull succeeds, `proposals.sql` and `proposals-sent-tracking.sql` are
captured inside the pulled migration and can be deleted — they were the manual,
run-by-hand scripts this workflow replaces. (The CLI ignores loose `.sql` files
in this directory; only `migrations/` and `seed.sql` are applied.)

## Rebuild a database from the migrations

```bash
# Local dev DB (Docker): apply every migration from scratch
supabase db reset

# A new hosted project: link to it, then push
supabase link --project-ref <new-ref>
supabase db push
```

## Make a future schema change

Don't edit tables by hand in the dashboard anymore — that's how the schema
drifted out of the repo in the first place. Instead:

```bash
# Author a migration, edit the generated SQL, then apply it
supabase migration new <short_name>
supabase db push        # to the linked remote, or `supabase db reset` locally
```

Or, if you changed a linked/branch database directly, capture the delta:

```bash
supabase db diff -f <short_name>
```
